PRテンプレートに沿ってDraft PRを作成・または更新し、checks監視とGeminiレビュー対応を行う。

## PR作成 or 更新

1. 以下のコマンドを並列実行してブランチ・コミット・変更内容を確認する
   ```sh
   git branch --show-current
   git log main..HEAD --oneline
   git diff main...HEAD --stat
   ```

1a. ブランチ名が**20文字を超える**場合、20文字以内に短縮したブランチ名にリネームする
   - `git branch -m <新しいブランチ名>` でリネームする
   - 既にリモートにpush済みの場合はリネームしない（そのまま進める）

2. `gh pr view` を実行してPRが存在するか確認する
   - 存在しない場合 → 新規作成フロー
   - 存在する場合 → 更新フロー（現在のPR本文を `gh pr view --json body -q .body` で取得してから更新内容を検討する）

3. `.github/pull_request_template.md` を読み込んでテンプレートの構成を把握する

4. 上記の情報から、テンプレートに沿ったPR本文を作成する
   - チェックシートは**すべてチェック済み（`[x]`）**にする
   - 背景と課題を簡潔に記述する
   - 実装内容を箇条書きで記述する
   - `## xxxの変更したUsecaseの影響範囲` はCIが記載する領域なのでセクション見出しのみ残して内容は記述しない
   - テンプレートにあるセクションはすべて含める（削除禁止）
   - テンプレートにある HTML コメント（`<!-- ... -->`）はそのまま残す（削除禁止）
   - セクション構造を変更しない（順序を維持）
   - 追加情報が必要な場合は、独立したセクションではなく実装内容に追記する
   - タイトルは **100文字以内**。要点をまとめて簡潔にする
   - 本文の強調（`**...**`）は要点のみに限定する。それ以外の箇所に強調を使わない
   - バッククォート（`` ` ``）は強調したいコードにのみ使う
   - 固有名詞の大文字小文字に注意する（例: `OmnibusCore`、`StartsAt` など）

5. Bash ツールで `mktemp -d` を実行して一時ディレクトリを作成し、返ってきたパスを記録する
   - Write ツールを使って `<一時ディレクトリ>/pr_body.md` にPR本文を書き出す

6. **必ずユーザーにPRのタイトルと本文を確認してもらう**
   - 一時ファイルのパスをユーザーに伝え、編集できることを案内する
   - ユーザーの承認を得るまで次のステップに進まない
   - 承認する場合は [ok] と入力するように案内する。それ以外は承認と認めない

7. ユーザーの承認後、まだpushしていない場合は `git push -u origin HEAD` を実行する

8. 承認されたPR本文で Draft PRを作成・または更新する
   - 新規作成: `gh pr create --draft --title "<タイトル>" --body-file $TMPDIR/pr_body.md`
   - 更新: `gh pr edit --title "<タイトル>" --body-file $TMPDIR/pr_body.md`

## PR作成後

以下の「checks対応」と「review対応」を**並列**で実行する。両方が完了したら終了する。

### checks対応

- `gh pr checks <PR番号> --watch` をバックグラウンドで実行する
- checksが失敗した場合はユーザーに報告する
- review対応で修正・pushが発生した場合、checksが再度走る。最新のchecksを監視し続ける

### review対応

`/gemini review` は1PRにつき**最大3回**まで。3回に達したらどんなときでも終了する。

実行回数を `gemini_review_count` としてカウントする（初期値0）。

#### 0. PRの完成度を自動判定する

`git diff main...HEAD` の内容から完成度（`pr_confidence`）を判定する。

**`pr_confidence = high` の条件（すべて満たす場合のみ）:**
- 変更ファイルが5個以下
- 変更が1〜2レイヤーに収まっている（domain, usecase, handler, infra 等）
- diff行数が少ない（追加+削除が100行以下）
- TODO/FIXME/HACKが差分にない

**上記を1つでも満たさない場合は `pr_confidence = low`**（安全側に倒す）。

#### 1. `/gemini review` をリクエストする

`gh pr comment <PR番号> --body "/gemini review"` でレビューをリクエストする。`gemini_review_count` をインクリメントする。

#### 2. レビューコメントを待つ

Geminiのレビューコメントが投稿されるまで待つ（`~/.claude/scripts/claude-pr-comments.sh` で確認）。
- `~/.claude/scripts/claude-pr-comments.sh` はGemini以外のユーザーのコメントも含めて未解決コメントをすべて返す

#### 3. 未解決コメントを分類する

未解決コメントがない場合 → 終了（レビュー完了）。

未解決コメントがある場合、まず各コメントを分類する:
- **褒め・情報提供・指摘なしのコメント** → 即resolveする（多数決不要）
- **修正を求める指摘コメント** → ステップ4へ

即resolve後、修正を求める指摘コメントが0件の場合:
- `pr_confidence = high` → **終了**（完成度が高く指摘もないため再レビュー不要）
- `pr_confidence = low` → ステップ1に戻る（見落としの可能性があるため再レビュー）

#### 4. 指摘コメントの妥当性を検討する

修正を求める指摘コメントについて妥当性を検討する（Gemini・他のユーザー問わず全コメントを対象にする）:
- 2つのエージェントを使って多数決で判断する
- 2つのエージェントの結果が一致した内容を採用する
- 一致しない場合は追加調査する
- 調査・検討してもわからないことがあればユーザーに質問する
- **絶対に曖昧な状態で進めない**

#### 5. 検討結果に基づいて処理する

- **妥当なコメントの場合**: 修正 → コミット → push
- **妥当じゃないコメントの場合**: PRレビュースレッドをresolveする（minimizeではない）
  - `gh api graphql` で `resolveReviewThread` mutation を使用する

#### 6. ループ継続判定

- `gemini_review_count` が3に達した場合 → 終了する
- 妥当じゃないコメントがこのレビューラウンドで**2つ以上**あった場合 → `gemini_review_count` が3未満ならもう1回だけ `/gemini review` する（ステップ1に戻る）。3に達していたら終了する
- 上記以外 → ステップ1に戻る

## 注意事項

- ユーザーの承認なしにPRを作成・更新しない
- PRは必ずDraft状態で作成する
- ですます調にしない。する調にする
- ユーザーが一時ファイルを直接編集した場合は、その内容を尊重してそのまま使う
